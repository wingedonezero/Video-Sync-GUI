3. KEY FINDINGS (DVD/Interlaced/VFR/Telecine)
========================================================================

Content handling:
- Time-domain search (ms offsets) works for ALL content types
- Audio→video offset difference is always integer frames (0-3)
- Deinterlacing (bwdif) gives ZERO improvement for frame matching
- IVTC (VFM+VDecimate) is catastrophically WORSE — never use for matching
- Raw frames (no processing) work fine for finding correct offset
- No content type detection needed — FFMS2 gives raw frames regardless

FFMS2 behavior:
- Progressive: frames as-is
- Soft-telecine: ignores repeat-field flags, gives original ~24fps frames
  - real_fps may be non-standard (e.g., 24.405 instead of 23.976)
  - VideoReader does AssumeFPS to 23.976 for display, but real_fps is used
    for frame index math (matches actual frame count)
- Hard telecine/interlaced: raw frames still match between different encodes
- get_frame_index_for_time() uses _AbsoluteTime timestamps from FFMS2 for
  accurate time→frame mapping (handles VFR correctly)

Audio correlation:
- SCC must use same language (JPN→JPN); eng doesn't match well
- Most MPEG-2 DVDs are effectively CFR (33/34ms alternation, <0.5ms drift)
- VFR detection at 100ms threshold catches true VFR files
- Audio correlation is rock-solid (99%+ match quality) for these test files

SSIM implementations (both in frame_hashing.py):
- Windowed SSIM (scikit-image, use_global_ssim=False):
  - More accurate for same-source/progressive content
  - Produces distances 35-45 for cross-encode DVD content (too high!)
- Global SSIM (mean/variance, use_global_ssim=True):
  - Uses whole-image statistics instead of local windows
  - Tolerant of deinterlace artifacts, resolution differences, encoder differences
  - Produces much lower distances for cross-encode content
  - Auto-selected by double-probe when windowed baseline exceeds threshold

========================================================================
5. ALGORITHM FLOW (matcher.py)
========================================================================

1. Get audio correlation delay (pure_correlation_ms)
2. Open VideoReaders with raw frames (deinterlace="none", no IVTC)
3. Get real FPS from FFMS2 (real_fps for index math, fps for display)
4. Generate time-domain candidates: audio_delay ± N * source_frame_duration
5. Select 9 checkpoint times distributed across video
6. Double-probe adaptive threshold:
   a. Probe 3 points with windowed SSIM
   b. If baseline > threshold → re-probe with global SSIM
   c. Decision: use_global_ssim=True, threshold stays or widens modestly
7. For each candidate offset:
   - At each checkpoint: compare source frame at T with target at T+offset
   - Sequence verify: check N consecutive frames match (70% threshold)
   - Sequence step uses min(source_frame_dur, target_frame_dur)
   - Record seq_verified count and MSE
8. Select best: max(seq_verified, then lowest MSE as tiebreaker)
9. Snap to frame-aligned offset via calculate_subframe_offset()
10. Return final_offset = sub_frame_offset + global_shift

9. ARCHITECTURE NOTES
========================================================================

VideoReader (video_reader.py):
  - Wraps VapourSynth + FFMS2 for frame extraction
  - get_frame_index_for_time(): uses _AbsoluteTime from FFMS2 for VFR accuracy
  - real_fps: pre-AssumeFPS fps (matches actual frame count)
  - fps: post-AssumeFPS fps (used for display/standard calculations)
  - For soft-telecine: AssumeFPS relabels to 23.976, but real_fps stays ~24.4
  - content_type parameter removed from video-verified calls (not needed)
  - deinterlace/IVTC settings still used by subtitle-anchored sync mode

compare_frames (frame_hashing.py):
  - Returns (distance, is_match) tuple
  - distance: (1.0 - SSIM) * 100 for SSIM method (0=identical, 10=SSIM 0.90)
  - use_global_ssim parameter selects between windowed (scikit-image) and
    global (mean/variance) implementations
  - Global SSIM: less sensitive to local pixel differences, better for
    cross-encode/deinterlaced content

generate_time_candidates (candidates.py):
  - Generates offsets in ms: audio_delay + N * source_frame_duration
  - N ranges from -search_range to +search_range (default ±8)
  - Also includes audio_delay itself and 0ms as candidates

select_checkpoint_times (candidates.py):
  - Distributes N checkpoints across video duration
  - Excludes first/last 15% to avoid credits/intros

calculate_subframe_offset (offset.py):
  - Snaps time-domain winner to frame-aligned output
  - integer_frames * source_frame_duration_ms
  - Optionally uses PTS precision from match_details

11d. TEST: VideoTimestamps library for time-domain frame mapping
------------------------------------------------------------------------
File: test_videotimestamps.py
Location: /home/chaoz/Desktop/Makemkv/Test Files/Interlaced/
Library: https://github.com/moi15moi/VideoTimestamps (v1.1.0)

Purpose: Instead of sweeping frame index offsets and converting with
average frame durations (lossy for VFR), use the VideoTimestamps library
to do exact PTS-based frame↔time mapping.

API pattern:
  from video_timestamps import VideoTimestamps, TimeType
  from fractions import Fraction
  from pathlib import Path

  vts = VideoTimestamps.from_video_file(Path(video_path))
  # frame_to_time returns SECONDS as Fraction (not ms!)
  ref_time = vts.frame_to_time(frame_num, TimeType.EXACT)
  # time_to_frame takes seconds as Fraction, returns int frame
  frame = vts.time_to_frame(time_fraction, TimeType.START)
  # pts_list contains ms values, time_scale = 1000

  # For files where video isn't stream 0 (e.g. subtitle first):
  vts = VideoTimestamps.from_video_file(path, index=1)

Mapping pattern (ref frame → target frame with delay):
  delay_s = Fraction(int(delay_ms * 1000), 1000000)
  ref_time = vts_ref.frame_to_time(ref_frame, TimeType.EXACT)
  tgt_time = ref_time + delay_s  # positive = tgt ahead of ref
  tgt_frame = vts_tgt.time_to_frame(tgt_time, TimeType.START)

Sign convention: tgt_time = ref_time + delay
  Positive delay = target content is temporally ahead of reference
  (e.g., if S2 plays 1 second of black before S1 starts, delay = +1000ms)

Tested on confirmed-matching pairs:

  Starship Ep1 (29.970↔29.970 CFR, 0ms delay):
    300/300 matches — trivial case, validates the mapping is correct

  Starship Ep2 (29.970↔23.976 CFR, +1001ms delay):
    299/300 matches — cross-fps with exact Fraction arithmetic
    Initially got 37/300 due to two bugs:
      Bug 1: frame_to_time returns seconds, code divided by time_scale
              unnecessarily in some paths (unit confusion)
      Bug 2: delay sign was wrong (-1001 instead of +1001). In test_cross_fps
              convention, offset=-1001 meant S1 behind S2. In VideoTimestamps
              test with ref=S1, tgt=S2, since S2 is AHEAD, delay must be
              +1001 (tgt_time = ref_time + positive_delay)

  Blassreiter S1↔S2 (29.778 VFR ↔ 29.970 CFR, +1001ms delay):
    300/300 matches — VFR↔CFR with exact PTS mapping

Key finding: VideoTimestamps provides clean exact frame↔time mapping
using Fraction arithmetic. Properly handles VFR, different CFR rates,
and any FPS combination. The time-domain approach (map through real PTS)
is fundamentally correct for all cases unlike index-offset approaches
that drift with different frame rates.

11e. OVERALL FINDINGS FROM TESTING
------------------------------------------------------------------------

What works for frame matching (no deinterlace/IVTC needed):
  - Same-fps CFR ↔ CFR: phash perfect (ph=0)
  - Cross-fps CFR ↔ CFR: phash perfect with correct time-domain offset
  - Interlaced ↔ progressive (same master): phash works, comb filtered
    by 32x32 downscale
  - VFR ↔ CFR (same master): phash works with PTS-based mapping
  - Different resolution same master (720x480 ↔ 704x480): phash works
  - Different encode same master (different BD releases): app's SSIM
    verifier works with 9 checkpoints + voting (phash_exact=85/90)

What does NOT work:
  - True different master (DVD ↔ BD): no visual metric works. Different
    color grading, resolution, luma levels. Only audio correlation.

Audio correlation notes:
  - Must match language AND channel format (5.1↔5.1, stereo↔stereo)
  - Wrong track selection gives ~0.21 confidence vs 0.98 correct
  - First chunk can be noisy (silence/intro), "first stable" mode handles it
  - Correlation gets the delay within ~1 frame, video verifier refines

VideoTimestamps vs index offset:
  - Index offset grows over time with different FPS
  - PTS delta stays constant — this is what mkvmerge needs
  - VideoTimestamps provides exact PTS↔frame mapping via Fraction math
  - Avoids floating-point drift from average frame duration calculations

12b. SEQUENCE SLIDING: BREAKTHROUGH (100% accuracy)
------------------------------------------------------------------------

Approach (user's idea): Instead of matching individual frames, slide a
10-second SEQUENCE from Source 1 across a wider window in Source 2.
At each slide position, compute mean cosine similarity across all
aligned frame pairs. The position with the highest cumulative score
is the correct offset. Like visual cross-correlation.

Test: Outbreak Company EP3, Source 1 ↔ Source 2
  Same master, same fps (23.976↔23.976)
  10-second window, ±5s slide range, 9 positions (10%–90% of episode)

  ISC (256-dim):
    9/9 exact match (100%), 239/239 frame matches at every position
    Mean score: 0.9993, range [0.9987, 0.9996]
    ~23s per position

  DINOv3 ViT-B/16 (768-dim):
    9/9 exact match (100%), 239/239 frame matches at every position
    Mean score: 0.9989, range [0.9984, 0.9993]
    ~55s per position

Score landscape (DINOv3-B, 50% position) — perfect symmetric gradient:
  off= +15: 0.9880    off= +20: 0.9948    off= +23: 0.9992 ★ PEAK
  off= +16: 0.9894    off= +21: 0.9962    off= +24: 0.9977
  off= +17: 0.9909    off= +22: 0.9977    off= +25: 0.9962
  off= +18: 0.9923                         off= +26: 0.9948
  off= +19: 0.9936                         off= +31: 0.9880
Drops ~0.0014 per frame from peak, perfectly symmetric both sides.
Completely unambiguous — no false peaks anywhere in ±5s range.

How the sliding works:
  1. Extract ISC/DINOv3 features for every frame in a 10s source window
  2. Extract features for a wider target window (10s + 2×5s padding)
  3. Slide source sequence across target one frame at a time
  4. At each slide position p: pair src[i] with tgt[p+i], compute
     cosine similarity for each pair, take mean across all pairs
  5. Position with highest mean = correct offset
  Key function: slide_and_score() — normalize features, dot product

ISC is the better choice: 2.3× faster, slightly higher scores, 256-dim
vs 768-dim features. ISC = Image Similarity Challenge (Meta ISC21),
designed for near-duplicate image detection — exactly this use case.

Test file: test_ep3_sequence_slide_v2.py

Sign convention note: The sliding finds that source frame N's content
appears at target frame N+23 (offset = tgt_frame - src_frame = +23).
The audio correlation reports -959ms / -23 frames using the opposite
convention (delay to apply). Same information, different sign.

12e. KEY TAKEAWAYS

1. Sequence sliding works perfectly for same-master same-fps content.
   ISC is the recommended model (faster, higher scores, simpler).

2. Per-frame neural matching is fundamentally wrong for this use case.
   All 9+ models tested failed. The problem is the content (static
   anime scenes where neighboring frames are nearly identical), not
   the models.

   interlaced 29.970fps deinterlaced with bwdif → 8/9 exact, 0.995+

   or frame count — just cleans combing artifacts in place.

   009-1 S1↔S3 (29.970↔23.976, 720×480↔704×384): scores near zero.
   Starship Operators S1↔S2 (29.970↔24.405): same failure.
   Must fall back to audio-correlation-only for these pairs.

5. For production, video-verified mode needs to detect pair type:
   - Same fps (or close): use ISC sequence sliding for verification
   - Cross fps / interlaced↔progressive with different resolution:
     skip visual verification, trust audio correlation only
   - Gate: if either source is MPEG2 interlaced/telecine AND fps
     differs significantly, use audio-only path. But pure progressive
     MPEG2 at same fps should still use visual verification.

6. For production: sequence sliding centered on the audio correlation
   offset would give frame-exact verification with ±a few frame search.

13. CRITICAL BUG: bwdif field=2 BOBS (doubles frame rate) — Session 4
========================================================================

DISCOVERY: bwdif field=2 does NOT mean "auto-detect field order".
It means BOB DEINTERLACE — doubling the frame rate!

  field=0: same rate, output bottom field
  field=1: same rate, output top field (CORRECT for TFF content)
  field=2: bob / double rate (WRONG — doubles frames and fps)

The bug: All session 3 interlaced tests used field=2, which turned
43,470 frames @ 29.970fps into 86,940 frames @ 59.940fps. This meant:
  - All frame indices were 2× the real value
  - +22f in bob space = +11f in real space
  - +734ms in bob space = +367ms in real space

Evidence:
  Before fix: s1_rgb.num_frames = 86,940, fps = 59.940 (DOUBLED!)
  After fix:  s1_rgb.num_frames = 43,470, fps = 29.970 (CORRECT)
  Ratio: 86940/43470 = 2.000, 22/11 = 2.000 — exactly 2×

Fix: Changed field=2 to field=1 in bwdif call:
  WRONG:   core.bwdif.Bwdif(clip_marked, field=2)  # BOB! doubles fps!
  CORRECT: core.bwdif.Bwdif(clip_marked, field=1)  # same rate, top field

NOTE: This bug also exists in the main app's VideoReader if bwdif
deinterlacing was being used with field=2. Check and fix there too.

NOTE: Section 12c-2 results were with the buggy field=2. The +22f
result was actually +11f. Section 12c-3 app method test was also
affected. Both corrected results are in Section 14 below.

15. SPEED OPTIMIZATIONS TESTED — Session 4

------------------------------------------------------------------------
15a. ISC Model Singleton Pattern
------------------------------------------------------------------------
Bug: Original test scripts did `del model` + `torch.cuda.empty_cache()`
after every position, then reloaded the model from disk for the next
position. With 9 positions, this wasted ~27 seconds on model I/O.

Fix: Global singleton pattern:
  _isc_model = None
  _isc_preprocess = None
  def get_isc_model(device):
      global _isc_model, _isc_preprocess
      if _isc_model is None:
          from isc_feature_extractor import create_model
          _isc_model, _isc_preprocess = create_model(
              weight_name='isc_ft_v107', device=str(device))
      return _isc_model, _isc_preprocess

This eliminates ~3s × 9 = 27s of redundant model loading.

------------------------------------------------------------------------
15b. 1-Second Window (10.6× speedup, but less robust)
------------------------------------------------------------------------
Test file: test_ep1_fast_slide.py (run with bob bug, offsets are 2×)

Config: 1s window (29 frames), ±1s slide, 9 positions, batch=32

  Result: 7/9 consensus at +22f (with bob bug)
  Two failures at static scenes — landscape completely flat within 0.001
  Time: 2.17s/position (vs ~23s for 10s window) = 10.6× speedup

  Tradeoff: 10× faster but vulnerable to static/low-motion scenes.
  Not reliable enough for production without scene activity detection.

15c. clip.frames() Prefetch (1.5× speedup, same accuracy)
------------------------------------------------------------------------
Test file: test_ep1_speed_compare.py

Two methods compared head-to-head:
  Method A (original): get_frame() → PIL Image → ISC preprocess → model
  Method B (optimized): clip.frames() → numpy → torch → GPU resize → model

Method B key changes:
  - VapourSynth clip.frames() prefetches internally (faster I/O)
  - Skip PIL Image creation (numpy → torch directly)
  - GPU resize via F.interpolate (instead of CPU PIL resize)
  - ImageNet normalize on GPU

  Method A: 143.4s total, ~16.0s/position
  Method B:  97.7s total, ~10.9s/position
  Speedup: 1.5× (46% faster)

  Accuracy: 9/9 offset agreement (identical offsets at every position)
  Scores: Method B slightly lower (0.9889 vs 0.9956 mean) due to
  GPU bilinear resize vs PIL Lanczos resize — doesn't affect offset finding

15d. Speed Summary
------------------------------------------------------------------------

Baseline (session 3, no optimizations): ~483s for 9 positions (ISC)
  - Model reloaded 9 times
  - get_frame() one at a time
  - PIL preprocessing on CPU

With singleton + clip.frames() + batch GPU: ~98s (4.9× faster)
With 1s windows on top of that: ~20s total (24× faster, but 7/9)

For production, recommended: singleton + clip.frames() + 10s windows
gives reliable 9/9 in ~100s. Can reduce to 3-5 positions if only
verifying audio correlation (not blind discovery).

Full pipeline:

  preprocessing.run_per_source_preprocessing()
    For each source_key ("Source 2", "Source 3", ...):
      _calculate_offset_for_method()
        if method == "neural":
          _run_neural_matching()
            1. Dump settings to config.json (temp)
            2. Spawn subprocess:
               python -m ...neural_subprocess
                 --source-video <path>
                 --target-video <path>
                 --total-delay-ms=<float>     ← = syntax for negative numbers
                 --global-shift-ms=<float>    ← = syntax for negative numbers
                 --config-json <config.json>
                 --output-json <result.json>
                 --temp-dir <job_temp>
                 [--debug-output-dir <dir>]
                 [--source-key "Source 2"]
            3. Forward subprocess stdout to runner._log_message()
            4. Parse __VSG_NEURAL_JSON__ prefix for result JSON
            5. Load result.json → (final_offset_ms, details)
      Store result in ctx.video_verified_sources[source_key]
      Store delay in ctx.subtitle_delays_ms[source_key]

  sync_dispatcher.apply_sync_mode()          ← called per subtitle track
    if source_key in ctx.video_verified_sources:
      _apply_cached_video_verified()          ← uses cached result directly
      (checks cached.get("fallback") for audio correlation fallback)
    else:
      run full video-verified plugin          ← only if no preprocessing ran

Subprocess internal flow (neural_subprocess.py):
  1. Parse args → load settings from config JSON
  2. Import neural_matcher.calculate_neural_verified_offset()
  3. Open videos via VapourSynth + FFMS2
  4. FPS check (ratio > 1.01 → fallback to audio correlation)
  5. Load ISC model (singleton, GPU)
  6. For each of 9 positions (10%-90% of video):
     - Extract source features: window_sec * fps frames → batch ISC → 256-dim
     - Extract target features: same + ±slide_range_sec padding
     - Slide source window across target: cosine similarity at each position
     - Best slide position = offset for this checkpoint
  7. Build consensus: most common offset across all positions
  8. Confidence: HIGH (≥0.9 consensus + ≥0.98 score), MEDIUM, LOW
  9. Write debug report (if enabled)
  10. Print __VSG_NEURAL_JSON__ {success: true, json_path: "..."}

Communication protocol:
  - Config in:   JSON file (settings + video paths)
  - Logs out:    stdout lines (forwarded by parent via readline)
  - Result flag: stdout line prefixed with __VSG_NEURAL_JSON__
  - Result data: separate JSON file (detailed results)
  - Errors:      __VSG_NEURAL_JSON__ {success: false, error: "..."}

Root cause: Audio correlation FFT produces floating-point noise near
zero for Source 3 (which had near-zero real delay):
  total_delay_ms = -3.3603082538746776e-09

When passed as separate list elements ["--total-delay-ms", "-3.36e-09"],
argparse interprets the leading '-' in "-3.36e-09" as the start of a
new flag, leaving --total-delay-ms with no value argument.

Why Source 2 worked: had a real negative delay like -21.333ms, which
argparse handles fine (it only fails with scientific notation like -3e-9
where the format looks flag-like).

Fix: Use --flag=value syntax (single list element) for all numeric args:
  BEFORE: "--total-delay-ms", str(total_delay_ms)     ← two elements
  AFTER:  f"--total-delay-ms={total_delay_ms}"         ← one element

The noise itself (-3.36e-09ms = 0.000000003ms) is harmless — 9 orders
of magnitude below anything meaningful. It comes from binary float
representation + FFT accumulation of tiny errors. Not worth clamping.

------------------------------------------------------------------------
19b. FALLBACK TO CLASSIC INSTEAD OF AUDIO CORRELATION (a5dfa59b)
------------------------------------------------------------------------

Symptom: When neural fails for Source 3, each subtitle track from
Source 3 runs full classic video-verified frame matching from scratch
instead of falling back to the audio correlation delay.

Root cause: In preprocessing.py, when neural matching returned None
(failure), corrected_delay_ms was None, so source was NOT stored in
ctx.video_verified_sources. In sync_dispatcher.py:
  if source_key in ctx.video_verified_sources:  ← False for Source 3
    _apply_cached_video_verified()               ← skipped
  ...
  # Falls through to NORMAL PATH → runs classic plugin per-track

This caused:
  - Re-running expensive frame matching per subtitle track
  - Using the wrong method (classic, not neural → should be audio)
  - Redundant work (preprocessing already determined matching failed)

Fix: When neural fails, store a fallback entry:
  ctx.video_verified_sources[source_key] = {
      "original_delay_ms": original_delay,
      "corrected_delay_ms": original_delay,  # audio correlation delay
      "details": details,
      "fallback": True,                      # sentinel flag
  }
  ctx.subtitle_delays_ms[source_key] = original_delay

sync_dispatcher._apply_cached_video_verified() checks:
  if cached.get("fallback"):
      # Log "using audio correlation fallback" and apply directly
  else:
      # Log "using pre-computed video-verified delay"

Both the normal failure path (frame matching returns None) AND the
exception path now store this fallback entry.

Important: Each source pair is INDEPENDENT. Source 2 neural succeeding
and Source 3 neural failing results in:
  Source 2 tracks → use neural-corrected delay from cache
  Source 3 tracks → use audio correlation fallback from cache
No cross-contamination between sources.
